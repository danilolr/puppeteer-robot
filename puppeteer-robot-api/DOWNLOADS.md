# Download Handling Notes

This document records the current download strategy and the cases that still need a different implementation.

## Current Model: `downloadUrl(url)`

The current command helper is:

```js
const file = await downloadUrl(url)
```

This model works well when the command can discover the final HTTP/HTTPS file URL from the current page.

Example:

```html
<a href="https://example.com/report.pdf">Report</a>
```

The command can read the `href` and ask the API server to download the file directly. The backend sends the current page cookies, user agent, and referer, which helps with files that depend on the active browser session.

This avoids relying on the browser's internal PDF viewer. If Chromium opens the PDF in its built-in viewer instead of emitting a traditional download event, `downloadUrl(url)` still works because it does not click the link.

## Limitation

`downloadUrl(url)` requires a URL known by the command.

It does not cover every real-world download flow. Some pages generate files as the result of a browser action, and the useful URL may not be visible in the DOM before the action happens.

## Cases That Need Another Model

### Button Opens a New Tab

Example:

```js
window.open('/reports/123/pdf')
```

The page has a button such as "Generate report". The URL only appears after clicking the button, possibly in a new tab.

Possible treatment:

- listen for new pages/popups;
- inspect the new page URL;
- if it is an HTTP/HTTPS URL that returns a file, save it or pass it through the download pipeline.

### Button Uses `fetch` or XHR

Example:

```js
const response = await fetch('/api/reports', { method: 'POST' })
const blob = await response.blob()
```

The page may call an API and receive the PDF as `application/pdf`.

Possible treatment:

- observe network responses while the action runs;
- detect file-like responses by `Content-Type` or `Content-Disposition`;
- persist the response body as a downloaded file.

### POST Generates the File

Some reports are generated by a POST request with body, headers, CSRF tokens, cookies, or other transient state.

In this case, repeating the URL later may not work. The best option is to capture the actual browser response caused by the original action.

Possible treatment:

- capture the request/response pair during the action;
- save the response body directly;
- record useful metadata such as method, URL, status, content type, and suggested file name.

### Blob URL Opens in the PDF Viewer

Example:

```js
const blob = await response.blob()
const url = URL.createObjectURL(blob)
window.open(url)
```

The new tab may show a `blob:` URL, not a reusable HTTP URL.

Possible treatment:

- capture the network response that produced the blob;
- or instrument browser APIs such as `Blob` and `URL.createObjectURL`;
- save the binary content before it becomes only a `blob:` viewer URL.

## Proposed Future Helper

A future command helper could look like this:

```js
const file = await captureFileFromAction(async () => {
  await page.click('#generate-report')
})

return {
  ok: true,
  file
}
```

The helper would execute the action while observing browser activity and save the first matching file response.

Potential detection rules:

- response header `Content-Type: application/pdf`;
- response header `Content-Disposition: attachment`;
- new tab opened with a PDF response;
- navigation to a PDF handled by Chromium's internal viewer;
- XHR/fetch response with file content;
- blob URL created from a captured response.

## Summary

Use `downloadUrl(url)` when the file URL is already known.

Use a future action-capture helper when the file is produced by clicking a button, submitting a form, opening a popup, receiving a POST response, or creating a browser `blob:` URL.
